## Orchestration explicite

Chaque `AskResponse` contient maintenant une trace de pipeline.

Noeuds actuels:

- `receive_question`;
- `route_intent`;
- `clarify`;
- `generate_sql`;
- `validate_sql`;
- `execute_readonly`;
- `compose_answer`.

Statuts possibles:

- `done`;
- `waiting`;
- `skipped`;
- `blocked`.

Cette trace permet d'expliquer pourquoi une reponse est produite, pourquoi une question demande clarification, ou pourquoi une execution est bloquee.

## Clarifications guidees

Quand aucune intention n'est assez fiable, FabriQ renvoie:

- un message de clarification;
- une liste `clarification_options`;
- pour chaque option: `intent_id`, label, question exemple et raison de suggestion.

L'interface affiche ces options sous forme de boutons. Un clic relance l'analyse avec la question exemple correspondante.

## Variables d'environnement

Par defaut:

```bash
FABRIQ_LLM_PROVIDER=disabled
```

Activation Ollama:

```bash
$env:FABRIQ_LLM_PROVIDER="ollama"
$env:FABRIQ_OLLAMA_URL="http://127.0.0.1:11434"
$env:FABRIQ_OLLAMA_MODEL="llama3.1"
$env:FABRIQ_LLM_TIMEOUT_SECONDS="60"
```

Le backend interroge `POST /api/generate` avec `format=json` et temperature `0`.

## Regle de securite

Le LLM n'a pas le droit de produire une requete SQL. Il ne fait qu'une proposition d'intention:

```json
{
  "intent_id": "supplier_delays",
  "confidence": 0.92,
  "reason": "The question asks about delivery delays."
}
```

Si l'intention n'appartient pas au catalogue ou si la confiance est faible, FabriQ demande une clarification.
